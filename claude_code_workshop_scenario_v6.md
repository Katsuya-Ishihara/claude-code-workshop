# Claude Code 活用ワークショップ シナリオ台本
**〜 チーム向け Todo タスク管理アプリで Claude 駆動開発を体感する 〜**

---

## このワークショップについて

本ワークショップでは、**AI コーディングツールとして Claude Code を採用**します。

従来の AI コーディング支援（GitHub Copilot に代表されるコード補完型）では、
「速くコードを書く」ことは実現できても、**「開発プロセスそのものを自動化する」**ことはできませんでした。

Claude Code は単なるコード補完にとどまらず、設計・Issue管理・実装・レビュー・PR作成までを
**エージェントとして自律的に実行**します。
本ワークショップでは、その違いをリアルタイムで体感することを目的とします。

---

## ワークショップの目的

**なぜ Claude Code を選ぶのか** ― GitHub Copilot との根本的な違いを実証する。

| | GitHub Copilot | **Claude Code（本ワークショップで採用）** |
|---|---|---|
| 役割 | コードを速く書く補助 | **開発プロセスごと代行** |
| 操作単位 | 行・関数 | **Issue・機能** |
| 人間の役割 | コードを書く人 | **設計・判断をする人** |
| 品質担保 | 人間がレビューする | **サブエージェントが客観レビュー＋人間が最終承認** |
| レビュー | 人間が全て実施 | **別コンテキストのサブエージェントが機械的チェック** |

> **採用理由まとめ**
> - 要件定義書を起点に、設計 → Issue登録 → 実装 → レビュー → PR 作成を **一気通貫で自動化**できる
> - コーディングガイドライン・セキュアコーディングの **機械的チェックを Claude が担当**する
> - 開発者は「コードを書く人」から **「設計と判断をする人」** へシフトできる

---

## 開発の全体フロー

本ワークショップでは、要件定義書を起点に以下の流れで開発を進めます。

```
Phase 0  👤 要件定義書の作成（人間が「何を作るか」を決める）
  ↓
Phase 1  🤖 CLAUDE.md・ガイドライン生成 → 👤 承認
  ↓
Phase 2  🤖 設計書生成（DB・API・画面）→ 👤 レビュー・承認
  ↓
Phase 3  🤖 タスク分解・Issue 自動登録 → 👤 内容確認
  ↓
Phase 4  🤖 テストファースト実装・コミット（Coding Agent）
  ↓
Phase 5  🤖 PR 作成 → 🤖 サブエージェントが客観レビュー → 👤 最終承認・マージ
  ↓
Phase 6  🤖 CI/CD 自動実行
```

### サブエージェントによるコードレビュー（Phase 5 の核心）

Claude Code には**サブエージェント**機能があります。
メインのエージェント（Coding Agent）が、別コンテキストで動作する独立したエージェントを
自律的に起動できる仕組みです。

```
Coding Agent（実装を担当）
  │
  │ PR 作成後、サブエージェントを起動
  ↓
Review Agent（サブエージェント）
  ├─ 実装時のコンテキストを一切持たない
  ├─ コードとガイドラインだけを見て機械的にチェック
  └─ 「自分で書いたコードを自分でレビュー」する構造的限界を回避
```

> **なぜサブエージェントが画期的なのか**
>
> 同じ Claude に「レビューして」と頼んでも、自分が書いた前提に引きずられて客観性が下がります。
> サブエージェントは**実装時の文脈を一切持たない別プロセス**なので、
> コードとガイドラインだけを見て判定できます。
> これは人間のレビュープロセス（書いた本人以外がレビューする）を AI で再現したものです。

│                                                               │
│  ✅ Phase 0：要件定義書の作成（唯一の一次情報。ここが全ての起点）│

> **なぜ人間の最終承認が必要か**

> コードは正しく書けていても「作るものが間違っている」ケースは Claude には検出できません。
> **AIは速度と網羅性を上げる手段であり、最終的な品質の責任は人間が持ちます。**

# チーム向け Todo タスク管理アプリ 要件定義書

## システム概要
チームメンバーが共同でタスクを管理するための Web アプリケーション。
Todo の作成・担当者アサイン・期限管理・進捗確認を一元化し、
口頭やチャットに頼ったタスク管理からの脱却を目的とする。

## 技術スタック
- フロントエンド：Blazor WASM（.NET 8）／レスポンシブデザイン必須
- バックエンド：ASP.NET Core Web API
- データベース：SQLite（ワークショップでの手軽さを優先。環境構築不要で即座に利用可能）
- 認証：JWT

## 開発フェーズ
- フェーズ1（v1.0.0）：一般メンバー向け Must 機能のみ。AWS EC2 にデプロイ。
- フェーズ2（v2.0.0）：管理者機能・Want 機能を追加。AWS ECR（コンテナ化）＋ GitHub Actions。

## 主要機能
| フェーズ | 対象ユーザー | 機能 |
|----------|-------------|------|
| フェーズ1 | 一般メンバー | ユーザー登録・ログイン |
| フェーズ1 | 一般メンバー | Todo の作成・編集・削除・完了 |
| フェーズ1 | 一般メンバー | 担当者のアサイン |
| フェーズ1 | 一般メンバー | 期限・優先度の設定 |
| フェーズ2 | 一般メンバー | 進捗率の入力（0〜100%） |
| フェーズ2 | 管理者 | ユーザー管理（追加・削除・権限変更） |
| フェーズ2 | 管理者 | カテゴリ管理 |
| フェーズ2 | 一般メンバー | カテゴリ・タグによるフィルタリング |
| フェーズ2 | 一般メンバー | 通知機能 |
| フェーズ2 | 管理者 | ダッシュボード（進捗サマリー） |

## Phase 1：CLAUDE.md・ガイドラインのセットアップ【🤖＋👤】

**目的：** Claude Code が「このプロジェクトのルール」を理解した状態にする

> 🤖 Claude が生成 → 👤 人間が内容を確認・承認する。
> ここで決めた内容が全フェーズの品質基準になる。

### CLAUDE.md とは

CLAUDE.md は **Claude Code 専用の設定ファイル**です。
プロジェクトのルートに置くと、Claude Code が起動時に自動で読み込みます。

```
人間の開発者  →  README.md を読んでプロジェクトを理解する
Claude Code   →  CLAUDE.md を読んでプロジェクトを理解する
```

ここに「技術スタック」「コーディングルール」「コミット規約」などを書いておくと、
Claude は毎回の指示で繰り返し伝えなくても、プロジェクトのルールに従って動作します。
逆に言えば、**CLAUDE.md に書かれていないルールは Claude は知らない**ので、
ここの品質がプロジェクト全体の品質を左右します。

### 1-1. CLAUDE.md の生成

> **[ライブデモ：claude.ai のターミナルで実行]**

**📋 プロンプト（コピペして使用）：**
```
docs/要件定義書.md を読んで、Claude Code 向けの CLAUDE.md を生成してください。

含めるべき内容：
- プロジェクト概要と技術スタック（要件定義書から読み取ること）
- ディレクトリ構成
- 開発ルール（テストファースト徹底）
- コミットルール（Issue番号紐付け必須、Conventional Commits形式）
- コーディングガイドライン参照先（CODING_GUIDELINES.md）
- セキュアコーディングガイドライン参照先（SECURE_CODING_GUIDELINES.md）
- PR作成ルール
- エージェント向け指示（PR作成後に必ず Review Agent にレビュー依頼すること）
```

### 1-2. コーディングガイドライン生成

**📋 プロンプト（コピペして使用）：**
```
.NET 8 / C# 12 向けのコーディングガイドライン（CODING_GUIDELINES.md）を
作成してください。

含めるべき項目：
- 命名規則（クラス・メソッド・変数・定数・インターフェース）
- 非同期処理のルール（async/await）
- null安全のルール（nullable reference types）
- DIコンテナの使用方針
- ログ出力の書き方
- コメントの書き方（XMLドキュメント）
- ファイル・クラスの責務分離ルール
各ルールには ✅ 良い例 と ❌ 悪い例 を必ず付けてください。
```

### 1-3. セキュアコーディングガイドライン生成

**📋 プロンプト（コピペして使用）：**
```
ASP.NET Core Web API 向けのセキュアコーディングガイドライン
（SECURE_CODING_GUIDELINES.md）を作成してください。

含めるべき項目：
- 入力バリデーション（SQLインジェクション・XSS対策）
- 認証・認可（JWT取り扱い）
- シークレット管理（ハードコード禁止）
- 依存関係の脆弱性管理
- エラーハンドリング（スタックトレースの外部露出禁止）
- ログに機密情報を出力しないルール
- CORS設定
各ルールには ✅ 良い例 と ❌ 悪い例 を必ず付けてください。
```

## Phase 2：設計資料の生成【🤖＋👤】

**目的：** 要件定義書を渡すだけで設計書が出来上がることを実演する

> 🤖 Claude が `docs/要件定義書.md` を読み取って設計書を生成する。
> 👤 人間が生成された設計書をレビューし、解釈ミス・抜け漏れがないか承認する。

### 2-1. DB設計書の生成

**📋 プロンプト（コピペして使用）：**
```
docs/要件定義書.md を読んで、
データベース設計書を docs/db_design.md として作成してください。

含めるべき内容：
- ER図（Mermaid記法）
- 各テーブル定義（カラム名・型・制約・説明）
- インデックス設計
- EF Core のエンティティクラス定義（C#コード）

データベースは SQLite を使用すること。
フェーズ1・フェーズ2の機能分割を考慮したテーブル設計にすること。
```

### 2-2. API設計書の生成

**📋 プロンプト（コピペして使用）：**
```
docs/要件定義書.md と docs/db_design.md を読んで、
REST API 設計書を docs/api_design.md として作成してください。

含めるべき内容：
- 全エンドポイント一覧（メソッド・パス・説明・対応フェーズ）
- 各エンドポイントのリクエスト・レスポンス（JSON例）
- 認証が必要なエンドポイントの明示（JWT）
- エラーレスポンス仕様
- OpenAPI (Swagger) 形式での記述も付けること

フェーズ1対象のエンドポイントと
フェーズ2対象のエンドポイントを明示して分類すること。
```

> 💡 **Tips：この OpenAPI 設計が実装コストを大幅に下げる**
> API設計書に含まれる OpenAPI（YAML）仕様は、クライアントコードの自動生成・
> バリデーションの自動化・フロントとバックの並行開発を可能にします。
> 「Phase 2 で丁寧に設計した分が、Phase 4 で一気に回収できる」のが
> OpenAPI 駆動開発の本質です。
> → [詳細：OpenAPI駆動開発と省力化のしくみ](./docs/tips/openapi_driven_dev.md)

### 2-3. 画面設計書の生成

**目的：** Razor コンポーネントの単位を確定させる。これが Phase 3 のタスク分解の粒度の基準になる。

> 👤 **人間の作業：** 生成された画面設計を確認し、画面数・コンポーネント構成が妥当かを承認する。
> ここで確定したコンポーネント一覧が、そのままフロントエンドの Issue 単位になる。

**📋 プロンプト（コピペして使用）：**
```
docs/要件定義書.md と docs/api_design.md を読んで、
Blazor WASM の画面設計書を docs/screen_design.md として作成してください。

含めるべき内容：

1. 画面一覧（フェーズ・画面名・URL・説明）

2. Razor コンポーネント一覧
   以下の分類で全コンポーネントを列挙すること：
   - Pages/      ルーティング対象のページ（@page ディレクティブあり）
   - Components/ ページ内で使う再利用コンポーネント
   - Shared/     レイアウト・ナビゲーションなど共通コンポーネント
   各コンポーネントには以下を含める：
   - ファイル名（例：TodoList.razor）
   - 責務（何を表示・操作するか）
   - 使用するAPIエンドポイント
   - 対応フェーズ（v1.0.0 or v2.0.0）

3. APIクライアント一覧（Services/ 配下）
   - ファイル名（例：TodoApiClient.cs）
   - 呼び出すエンドポイント一覧

4. 画面遷移図（Mermaid記法）

フェーズ1・フェーズ2のコンポーネントを明示して分類すること。
```

## Phase 3：タスク分解 → Issue 自動登録【🤖＋👤】

**目的：** 設計書→Issueの自動変換を実演。フェーズ別・レイヤー別・Razor単位での粒度を実演。

> 👤 人間の作業：登録されたIssueの一覧を確認し、粒度・優先順位・依存関係に抜け漏れがないかを確認する。

### タスク粒度の考え方

```
バックエンド   → APIエンドポイント 1本 ＋ テスト = 1 Issue
フロントエンド → Razor コンポーネント 1個         = 1 Issue
               → APIクライアント（Services/）1ファイル = 1 Issue
インフラ       → EF Core マイグレーション 1本       = 1 Issue

依存関係の原則：
  バックエンドの Issue → フロントエンドの Issue
  （APIが先に完成していないと、Razor の実装・テストができない）
```

### 3-1. タスク分解

**📋 プロンプト（コピペして使用）：**
```
docs/要件定義書.md・docs/db_design.md・docs/api_design.md・
docs/screen_design.md を読んで、開発タスクを分解してください。

【粒度のルール】
- 1 Issue = 1〜2時間で完了できる粒度
- バックエンド：APIエンドポイント 1本 ＋ xUnit テスト = 1 Issue
- フロントエンド：Razor コンポーネント 1個 = 1 Issue
                 APIクライアント（Services/）1ファイル = 1 Issue
- インフラ：EF Core マイグレーション 1本 = 1 Issue

【各Issueに含める情報】
- タイトル（feat/fix/chore: 形式）
- 実装内容の説明
- 完了条件（Definition of Done）
- 依存するIssue（バックエンドのIssueが先行すること）
- 並行実行可否
- 対応フェーズ（v1.0.0 or v2.0.0）

【分類】
- backend / frontend / infra に分類して出力

結果をMarkdownの表形式で出力してください。
```

### 3-2. GitHub Issue の一括登録

**📋 プロンプト（コピペして使用）：**
```
先ほどのタスク一覧を GitHub Issue として登録してください。

gh コマンドを使って全Issueを作成します。
各Issueには：
- 適切なラベル（backend / frontend / infra / test）
- マイルストーン：フェーズ1のIssueは「v1.0.0」、フェーズ2は「v2.0.0」
- 本文に完了条件を含める

まず gh auth status で認証確認してから実行してください。
```

### 3-3. git worktree の展開

**📋 プロンプト（コピペして使用）：**
```
v1.0.0 のIssueの中から並行実行可能なものを確認して、
git worktree を使った並行開発環境を準備してください。

- 並行実行できるIssueを3つ選ぶ
- それぞれに対して worktree と feature ブランチを作成
- 各 worktree のパスとブランチ名の一覧を表示

例：
  ../todo-app-issue-3  →  feature/issue-3-user-auth
  ../todo-app-issue-5  →  feature/issue-5-todo-crud
  ../todo-app-issue-8  →  feature/issue-8-category-api
```

## Phase 4：テストファーストで実装・コミット【🤖】

**目的：** 「テストを書いてから実装する」を Claude が自律的に実行することを実演

> 🤖 Claude が全て担当するフェーズ。
> コミットは作業の区切りごとに自由に行う（品質ゲートはPRで担保）。

> 💡 **Tips：Phase 2 の設計資産を最大活用する**

### 4-1. Issue を読んでテストから書く

**📋 プロンプト（各 worktree のターミナルで実行）：**

GitHub Issue #[番号] を読んで、テストファーストで実装してください。

手順を厳守すること：

2. まずテストコードを書く（xUnit使用）

7. 全テストが通ることを確認する
8. 作業の区切りごとに git commit する
9. 実装完了後に PR を作成する（次のPhaseへ）

テストは src/TodoApp.Tests/ に作成すること。
CODING_GUIDELINES.md・SECURE_CODING_GUIDELINES.md・

## Phase 5：PR作成 → Claude による機械的品質チェック → 人間承認【🤖＋👤】

**目的：** PRをトリガーにClaudeが機械的チェックを自動実行し、人間が最終承認することを実演

### フロー全体像

```
Claude（実装完了）
  ↓
PR 作成（Claude）
  ↓
Review Agent による機械的チェック（Claude）← ここまで自動
  ├─ NG → Coding Agent に指摘を返す → 修正 → 再チェック
  │        ※ 3回NGなら人間にエスカレーション
  └─ OK → レビュー結果をPRコメントに投稿
              ↓
         👤 人間による最終レビュー・マージ承認 ← ここは必ず人間
              ├─ 承認 → マージ → Issue 自動クローズ
              └─ 差し戻し → Claude が修正 → 再チェックループへ
```

### 5-1. PR の自動作成

**📋 プロンプト（コピペして使用）：**
```
feature/issue-[番号] ブランチの実装が完了しました。
GitHub Pull Request を作成してください。

PR の内容：
- タイトル：Issue タイトルを元に生成
- 本文：
  - 変更内容のサマリー
  - テスト結果（dotnet test の出力）
  - APIの場合はcurlコマンド例
  - 関連Issue番号（Closes #XX）
- ラベル：適切なものを付ける
- レビュワー：[担当者名] をアサイン

gh pr create コマンドを使って実行してください。
```

### 5-2. Review Agent による機械的チェック

**📋 Review Agent のプロンプト（CLAUDE.md に記載しておく）：**
```
[Review Agent として動作してください]

あなたはコードレビュー専門のエージェントです。
ガイドラインに照らして機械的に判定してください。

レビュー対象：このPRで変更されたファイル全て（git diff main）

【Claudeが判定するチェック項目】
1. CODING_GUIDELINES.md の全ルールに準拠しているか
   - 命名規則（クラス・メソッド・変数・定数）
   - async/await の正しい使い方
   - XMLドキュメントコメントの有無
   - ファイル・クラスの責務分離

2. SECURE_CODING_GUIDELINES.md の全ルールに準拠しているか
   - ハードコードされたシークレットがないか
   - SQLインジェクション・XSSの典型パターンがないか
   - JWT取り扱いの既知ミスがないか
   - スタックトレースの外部露出がないか

3. テストコードが存在するか（テストなし実装はNG）

4. テストが全て通るか（dotnet test を実行して確認）

5. TODO/FIXME コメントが残っていないか

【出力形式】
- 判定：OK or NG
- NGの場合：指摘事項（ファイル名・行番号・理由）を箇条書きで
- OKの場合：以下を明記する
  「機械的チェック完了。設計の妥当性・ビジネスロジックの
   正しさ・アーキテクチャの判断は人間レビュワーにお任せします。」

【注意】以下は判断できないため、チェック対象外とする：
- 設計の意図が要件と合っているか
- アーキテクチャとして将来の拡張に耐えるか
- テスト自体が正しいか
- ビジネスロジックの正しさ
```

### 5-3. 人間による最終レビュー（必須）

```
👤 人間がレビューすべき観点：

□ 要件定義書の意図通りに実装されているか
□ アーキテクチャとして妥当か（将来の拡張・保守性）
□ ビジネスロジックが仕様書通りに動くか
□ テスト自体が正しいか（テストはパスしているが、テストが間違っていないか）
□ フェーズ1スコープに収まっているか（フェーズ2の機能が混入していないか）
```

## Phase 6：CI/CD パイプライン（GitHub Actions）【🤖】

**目的：** PRをトリガーに自動テスト・品質チェックが走ることを実演

### 6-1. GitHub Actions ワークフローの生成

**📋 プロンプト（コピペして使用）：**
```
.github/workflows/ に以下のGitHub Actions ワークフローを作成してください。

トリガー：Pull Request 作成・更新時

ジョブ：
1. build-and-test
   - dotnet restore
   - dotnet build --no-restore
   - dotnet test --no-build --verbosity normal
   - カバレッジレポート生成（coverlet）

2. code-quality
   - dotnet format --verify-no-changes（フォーマットチェック）
   - Roslyn アナライザーによる静的解析

3. security-scan
   - dotnet list package --vulnerable（脆弱な依存パッケージ検出）

全ジョブが通らないとマージ不可の設定にすること。
```

## Development Rules

### 要件定義書の扱い
docs/要件定義書.md は人間が作成した一次情報です。
Claude は読み取るだけで、編集・上書きしてはいけません。

### テストファースト（必須）
1. テストを書く → 2. RED確認 → 3. 実装 → 4. GREEN確認 → 5. リファクタ

### コミットルール
- Conventional Commits 形式: feat/fix/docs/test/chore: 説明
- 必ず Issue 番号を含める: Closes #XX
- コミットは作業の区切りごとに自由に行う（WIPコミットもOK）

### Review Agent への依頼方法
PR作成後、以下をそのままプロンプトに貼り付けること：
「Review Agent として動作してください。CODING_GUIDELINES.md と
SECURE_CODING_GUIDELINES.md に基づき、このPRの変更内容を
機械的にレビューしてください。」

### PR ルール
- 必ず Issue に紐付ける（Closes #XX）
- テスト結果を本文に含める
- Claude の機械的チェック（OK）＋ 人間の最終承認、両方揃ってからマージする
- CI が全てグリーンになるまでマージしない
```
